# Direction — как поделиться этим с коллегами

> Набросок направления, не спецификация. Цель — прочувствовать, куда двигать
> проект, чтобы им могли пользоваться другие разработчики в своих проектах.

## Что сейчас есть

Локальный двухсервисный инструмент:

- **control-plane сервер** (Effect-TS, Node) — оборачивает `orbctl`, держит pty
  сессии Codex, отдаёт diff. См. `control-plane/server/src/`.
- **web-панель** (React 19 + Vite, xterm.js) — список агентов, терминалы,
  diff-вьювер, кнопки start/stop. См. `control-plane/web/src/`.

Главное препятствие для шеринга — **не упаковка, а захардкоженность под Shilo**.
Всё проектно-специфичное сидит в `control-plane/server/src/config.ts`:

```text
PREFIX        = "shilo-agent-"
BASE_MACHINE  = "shilo-agent-base"
REPO_DIR      = "~/projects/shilo-ai-mono"
HASURA_SERVICES / BACKEND_DEP_SERVICES  — конкретный стек Shilo
```

Поэтому: **сначала вынести «что за проект» в конфиг, потом думать про упаковку.**
Иначе любая Electron-сборка просто запакует хардкод.

## Три уровня «поделиться» — по возрастанию усилий

```diagram
╭───────────────────────────────────────────────────────────────╮
│ 1. Параметризовать конфиг   → коллега запускает у себя вручную  │
│ 2. Упаковать в 1 артефакт   → CLI `npx` (Electron — потом)     │
│ 3. project profiles + base  → работает в ЛЮБОМ репозитории     │
╰───────────────────────────────────────────────────────────────╯
```

## 1. Сначала — конфиг проекта (фундамент для всего)

Вынести из `config.ts` всё проектно-специфичное в декларативный файл, который
кладётся в репозиторий целевого проекта (по аналогии с `.devcontainer`).

### Важно: конфиг НЕ должен быть на TS/JS

Инструмент оркеструет **любой** проект — он может быть на Python, Go, Rust, Java.
Если конфиг сделать `.ts`, мы навязываем чужому репозиторию наш язык и тулчейн
(node_modules, tsx, типы). Это неправильно: конфиг описывает *оркестрацию VM*, а
не код проекта.

Поэтому — **язык-независимый декларативный формат**: YAML (или TOML/JSON).

- читается без рантайма любого языка;
- естественно ложится в чужой репо рядом с `docker-compose.yml`;
- легко коммитить и ревьюить.

Цена: нет автокомплита и нет «логики» в конфиге. Но логика тут и не нужна — это
плоский манифест. Если когда-то понадобится динамика, можно поддержать
*опциональный* `keenfleet.config.{ts,js}`, который просто резолвится в тот же
объект. Но дефолт и рекомендация — YAML.

```diagram
╭─ keenfleet.yml  (в репо целевого проекта) ─╮
│  base_machine:   myapp-agent-base          │
│  machine_prefix: myapp-agent-              │
│  repo_dir:       ~/projects/myapp          │
│  repo_slug:      org/myapp                 │
│  branch:         dev                        │
│  stack:                                      │
│    - postgres                               │
│    - redis                                  │
│    - hasura                                 │
│  setup:          # команды для base-VM      │
│    - corepack enable                        │
│    - pnpm install                           │
╰─────────────────────────────────────────────╯
```

Это разблокирует и пункт 3 (другие проекты), и делает любую упаковку осмысленной.

## 2. Упаковка: CLI сейчас, Electron потом

**Electron на старте скорее лишний.** Почему:

- Бэкенд *обязан* быть локальным процессом — ему нужен `orbctl`, `node-pty`,
  docker на хосте. Это уже «нативно».
- Аудитория — разработчики на macOS с OrbStack, им комфортен терминал.
- Самый дешёвый шеринг: `npx keen-fleet` → поднимает сервер + открывает браузер.
  Один пакет, ноль установки.

В проде: билдить web в статику и раздавать её тем же сервером (убрать
vite-proxy, который сейчас в `control-plane/web/vite.config.ts`).

**Electron оправдан позже**, когда захочется: иконку в доке/трее, нативное окно
без вкладки браузера, авто-апдейты, «продуктовый» вид для не-CLI коллег. Это
polish, не enabler. Electron сможет переиспользовать тот же сервер.

```text
Порядок:  CLI-пакет (npx)  →  потом Electron-обёртка вокруг того же сервера
```

## 3. Чтобы работало в чужих проектах

Каждому проекту нужна своя «золотая» base-VM. Нужен флоу:

- `keen-fleet init` — интерактивно создаёт `keenfleet.yml` + строит base-VM
  (clone Ubuntu → выполнить `setup` команды → freeze).
- панель читает конфиг из cwd → `machine_prefix` уникален на проект → агенты
  разных проектов не конфликтуют.
- задокументировать предусловия: OrbStack установлен, доступ к репо, секреты
  (напр. `SENTRY_AUTH_TOKEN` из `docs/handoffs/01-base.md`).

## Где живёт конфиг — не обязательно в репо

Класть `keenfleet.yml` в чужой репо часто не вариант:

- **«палево»**: команда/безопасники не хотят файла стороннего AI-инструмента в
  своём репозитории; видно, что используешь агентов; лишний diff в PR.
- **права**: ты можешь не иметь права коммитить такое в общий репо.
- **личные настройки**: пути, префиксы, ресурсы — индивидуальны, им не место в
  общем репо.

Поэтому конфиг должен поддерживать **несколько мест**, с приоритетом:

```diagram
╭──────────────────────────────────────────────────────────────╮
│ 1. ./keenfleet.yml            в репо, закоммичен (расшарен)    │
│ 2. ./keenfleet.local.yml      в репо, в .gitignore (личный)    │
│ 3. ~/.config/keen-fleet/<project>.yml   вне репо вообще        │
╰──────────────────────────────────────────────────────────────╯
```

Дефолт для «не хочу палиться» → вариант 3: конфиг лежит у тебя в `~/.config`,
репо вообще не трогается. Привязка к проекту — по пути репо или git-remote.

## Конфиг можно собирать через UI (а не писать руками)

Да, и это, наверное, лучший онбординг. Вместо «отредактируй YAML» —
мастер в панели / `keen-fleet init`, который:

- сам определяет, что может: `repo_slug` из `git remote`, текущую ветку,
  сервисы из найденного `docker-compose.yml`, путь репо из cwd;
- спрашивает только остальное (база VM, ресурсы, setup-команды);
- по итогу **генерит YAML сам** и кладёт туда, куда ты выбрал (репо / local /
  `~/.config`).

То есть YAML остаётся как формат хранения (читаемый, диффаемый, без рантайма), а
UI — просто удобный способ его собрать и редактировать. Одно другому не мешает.

## Полирепо (не у всех монорепо)

Сейчас модель жёстко **один агент = один репо**: `repo_dir`, `repo_slug`,
`branch` в единственном числе, а `runInRepo` всегда ходит в один путь. Для
полирепо это ломается: задача часто живёт сразу в нескольких репо
(frontend + backend + общие либы), и агенту нужны они все в одной VM.

Хорошая новость — слой VM от этого не меняется. Меняется только конфиг и
агрегация. Конфиг становится списком репо:

```diagram
╭─ keenfleet.yml (полирепо) ─────────────────────╮
│  workspace_dir: ~/work/myapp                    │
│  repos:                                          │
│    - { slug: org/api,      dir: api,    branch: dev } │
│    - { slug: org/web,      dir: web,    branch: dev } │
│    - { slug: org/shared,   dir: shared, branch: main }│
│  stack: [postgres, redis]                       │
│  setup: [ ... ]                                  │
╰─────────────────────────────────────────────────╯
```

Что тянет за собой:

- **base-VM** содержит все репо склонированными + общий dev-стек (он и так
  общий, просто шире).
- **diff/статус** агрегируются по всем репо (сейчас diff одного — стало бы
  «diff по каждому»). Вьюверу — группировка по репо.
- **координация веток**: одна логическая задача = согласованные ветки во всех
  репо. Можно по соглашению (одинаковое имя ветки) или явно в конфиге.
- **возврат результата**: push/PR не в один репо, а в несколько — усложняет
  доставку.

Вывод: полирепо — это не «другой инструмент», а обобщение `repo` →
`repos[]`. Стоит сразу заложить список (даже если у тебя один элемент), чтобы
потом не переделывать конфиг и API.

## Что ещё легко упустить (более крупные слепые зоны)

- **Секреты**. Агентам внутри VM нужны `.env`, токены (видел `SENTRY_AUTH_TOKEN`,
  Codex-авторизация). Конфиг **не должен** их хранить — только ссылки
  («возьми из `~/.keenfleet/secrets` / из 1Password / из env хоста»). Это
  отдельный механизм, не путать с проектным манифестом.
- **Codex/agent auth в каждой VM**. Каждой свежей VM нужен залогиненный агент
  или API-ключ — это часть bootstrap, легко забыть.
- **Ресурсы ноутбука**. N агентов = N VM + N стеков (Postgres/Redis/...). RAM и
  диск кончаются быстро. Нужны лимиты и видимость потребления в UI.
- **Протухание base-VM**. Зависимости в репо меняются → «золотая» база
  устаревает. Нужен флоу пересборки/обновления базы, иначе клоны ломаются.
- **Уборка мусора**. CoW-клоны и брошенные VM копят диск. Нужен GC/«удалить
  всех остановленных».
- **Возврат результата**. Как работа агента попадает обратно: push ветки, PR?
  Сейчас есть только diff-вьювер — это просмотр, а не доставка.
- **Платформа**. OrbStack — только macOS. Для коллег на Linux/Windows это стоп
  (можно абстрагировать слой VM, но это большой кусок).
- **Безопасность локального сервера**. Сервер дёргает host-команды; `/term`
  принимает `machine` из запроса (сейчас прикрыт `MACHINE_RE`). При шеринге это
  место надо держать строго — никаких произвольных команд из UI.

## Рекомендуемый порядок шагов

1. Параметризовать `config.ts` → внешний `keenfleet.yml` с несколькими местами
   хранения (репо / local / `~/.config`).
2. `init`/UI-мастер, который **сам собирает** конфиг (автодетект + вопросы).
3. Отдельный механизм секретов (ссылки, не значения).
4. Завернуть оба сервиса в один `npx`-запускаемый пакет; web билдить в статику,
   сервер её раздаёт.
5. Bootstrap base-VM в любом репо + флоу её обновления/уборки.
6. Только потом — Electron-обёртка для трея и «продуктового» вида.
